# Promotion Criteria

## ■ 基本基準

- 複数観測をまたいで再利用可能である
- 条件依存のメモではなく、運用や説明の骨格になっている
- 一段上の層へ置くことで責務が明確になる
- 上位層へ置いたとき、過剰な権威化より責務明確化の利益が大きい

## ■ 代表的な昇格

### `logs -> fragments`

- 同じパターンが複数回観測された
- 単発現象ではなく構造的傾向として扱える

### `logs -> tips`

- その日から使える小さな工夫がある
- 観測の中に再利用可能な軽量ハックが含まれている
- `playbook` にするほどではないが、`logs` に埋めるには惜しい

### `logs -> todo`

- 観測差分から次アクションが明確に出ている
- 実行キューとして持つ方が自然である
- 知識化より着手が重要である

### `tips -> fragments`

- 小技として使っていたが、実は構造仮説を含んでいる
- 実行知より先に概念整理した方が再利用しやすい
- tip の背景原理を切り出したくなった

### `fragments -> playbook`

- 仮説が現実の行動に直接接続できる
- 実際の場面で発動できる粒度になっている
- 反証可能性がある
- trigger / indicators / default_response / escalation の少なくとも骨格が見えている

#### 「即時再利用価値がある」の具体化

以下のうち2つ以上を満たすとき、
反復3回未満でも `playbook` 候補として扱ってよい。

- 明日から参照する具体手順に落ちている
- 安全性、摩擦低減、過負荷回避のいずれかに直結する
- 状態の見分け方と最初の応答がすでに書ける
- 単なる気づきではなく、やる / やらない の差が明確である

### `tips -> playbook`

通常は昇格より、
関連する `tips / todo / playbook` を相互参照で結ぶ方を優先する。
以下を満たし、
tip の背景原理を正本化したいときだけ候補とする。

- 小技が反復している
- 概念層の運用ルールとして書ける
- trigger / indicators / default_response の骨格が見えてきた
- 状態の見分け方が書ける
- 局所ハックではなく、判断原則として再利用されている

### `todo -> tips`

通常は、
todo を消し、
必要なやり方だけを tip として別保持する。

- 同様の todo が複数回出現している
- タスク自体より、やり方を残した方が再利用しやすい
- todo を保存し続けるより、軽量な運用知へ蒸留した方が自然である

### `tips -> todo`

通常は昇格ではなく、
tip を参照して今回の todo を起こすリンク運用として扱う。

- 小技を知っているだけでは足りず、具体行動として起こしたい
- `こうやると楽` を `今これをやる` に落としたい

### `playbook -> theory`

- 個別ルールを超えた抽象構造が見えてきた
- 外部読解に耐える言語化が必要になった
- 仮説的拡張として理論化する価値がある

#### `playbook -> theory/operational`

以下のいずれかを満たすとき候補とする。

- 複数 playbook の共通構造を説明する必要がある
- playbook と core のあいだの橋渡し説明が必要である
- ルールより抽象的だが、essay にするにはまだ早い

#### `theory/operational -> new essay`

以下をすべて満たすとき候補とする。

- 橋概念が複数の playbook / fragment を一本の原理で束ねられる
- その概念で OS の挙動や設計思想を一篇の論旨として説明できる
- 既存 essay への追記より、新しい essay として独立させた方が締まる
- 外から読んでも運用メモではなく論述として自立する

#### `theory/operational -> narratives`

以下のいずれかを満たすとき候補とする。

- 複数の operational / playbook を、論旨ではなく内部の読み筋として束ねたい
- `core` と後続理論の連続性を、定義や essay より前の段階で保持したい
- essay を編む前に、どの筋で読むかを内部で明示しておきたい

#### `narratives -> essay`

以下をすべて満たすとき候補とする。

- narrative の筋を外部読解に耐える論述へ再構成できる
- narrative 自体を読まなくても essay が一篇として自立する
- 内部の読み筋をそのまま出すのでなく、選別と圧縮が済んでいる

#### `new essay -> core候補`

以下をすべて満たすときだけ候補とする。

- essay の中核概念が OS 一般の前提に食い込んでいる
- playbook / theory の説明語ではなく、core にないと全体の意味が不自然になる
- 運用主体固有の条件や時期依存の流行語ではない
- 既存 core の定義だけでは繰り返し説明しきれない

### `experiments -> interfaces`

- 試行ではなく、他者へ渡せる形式として安定した
- 実験条件に依存しない再利用可能な手順や形式になった
- 外部出力としての表現が主目的になった

### `experiments -> meta`

- 実験対象そのものではなく、継続運用の管理方法になった
- 一度きりの接続試行ではなく、正本・同期・判定基準の管理が主題になった
- 実験層よりも運用仕様層に置くほうが責務が明確になった

### `experiments -> playbook`

- 外部試行で有効だった対応が、日常運用の即応ルールとして再利用可能になった
- 実験固有の前提を外しても発火条件と対応が維持できる

### `experiments -> theory`

- 試行結果から個別ケースを超えた抽象構造が見えた
- 応用可能性や説明責務が実験報告より理論記述に寄った
