# Layer Addition Criteria

## ■ 基本基準

- ある種類の記述が複数層にまたがって繰り返し迷子になる
- 既存層の定義を無理に広げないと置き場所が作れない
- 更新頻度や抽象度が既存層と明らかに違う

## ■ 追加時の必須条件

新しい層を追加するときは、
受け皿ディレクトリだけを先に作らない。
少なくとも同一サイクルで以下を揃える。

- その層の責務を定義する README
- その層へ何を入れ、何を入れないかの基準
- 既存層との境界
- `meta/layer_management/` 側の追加記録

さらに、
新層が複数ファイルや子ディレクトリを持ちうる場合は、
`meta/directory_management/` 側でも
ディレクトリ責務を定義するか、
少なくとも「いまは平置きでよい理由」を確認する。

## ■ 補足

新設前に、
`playbook/` や `theory/operational/` で吸収できないかを確認する。

新しい実験領域を切りたい場合も、
まず既存の `experiments/` 配下で責務分離できないかを確認する。

`archive/` を作りたくなった場合も、
まず Git 履歴だけで目的を満たせないかを確認する。

それでも不足する場合だけ、
次の条件を満たすなら薄い `archive/` 層を追加してよい。

- 現役運用から外れたが repo 内参照価値が高い
- 旧正本や履歴スナップショットとして残す理由が明確
- `experiments/` や `past_daily/` と責務がぶつからない
