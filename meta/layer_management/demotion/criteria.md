# Demotion Criteria

## ■ 基本基準

- 一般構造ではなく局所条件や局所仮説に寄っている
- 現在の層にあることで過剰な権威を帯びている
- 一段下の層へ戻すことで更新しやすくなる
- 戻した方が責務と更新頻度が一致する
- 「強そうに見える」より「どこで扱うと自然か」を優先する

## ■ 代表的な降格

### `theory -> fragments`

- 観測回数が少ない
- 反証可能性が高い
- 単発の強い体験に引っ張られている
- 外部説明可能に見えても、実際には局所仮説にとどまる

### `theory -> operational`

- 一篇の論述としてはまだ閉じていない
- 中核概念はあるが、反例・境界条件・射程がまだ粗い
- essay としてより、橋概念の育成層に戻した方が自然である
- 既存 essay に乗せると論述全体を緩める

### `theory/operational -> playbook`

- 抽象説明より、具体的な発火条件と応答に落とした方が再利用しやすい
- 共通構造と思っていたが、実際には特定場面の運用ルールにとどまる
- 複数 playbook を束ねるほどではなく、一つのルールへ圧縮できる

### `narratives -> theory/operational`

- 全体の読み筋というより、単一の橋概念説明へ圧縮した方が自然である
- narrative で束ねるには対象が少なく、個別の operational 文書で足りる
- `core narrative` や全体 narrative としての連続性がまだ立っていない

### `narratives -> context`

- 全体の連続性に見えて、実際には運用主体固有の履歴や条件に強く依存する
- 内部読み筋というより、個人的な自己解釈メモとして置く方が自然である
- narrative として置くと過剰な一貫性や神話性を帯びやすい

### `playbook -> fragments`

- trigger や indicators は書けるが、default_response がまだ試行段階である
- 再利用したい感触はあるが、実際の発動場面が足りない
- 即応ルールとして置くと権威が先行し、観測継続の方が自然である

### `playbook -> tips`

- 概念層としては重く、実際には小さな工夫として使われている
- 原理より軽量な再利用知として持つ方が自然である
- trigger は弱いが、その場で効くことは明確である

完全な置換より、
背景原理としての `playbook` を残し、
実装手段だけを `tips/` へ切り出す方が自然なことが多い。

### `fragments -> tips`

- 仮説の実行検証として、まず小さく回してみたい
- 概念整理より、軽量ハックとして使う方が今は自然である
- fragment の補強材料を取りたい

### `tips -> logs`

- 効いたかどうかがまだ曖昧である
- その日の出来事記録として持つ方が自然である
- 小技として固定するより観測継続が重要である

### `todo -> logs`

- 実行して差分が出た
- 実行途中で重要な観測が生じた
- 完了後に状態差分として残す価値がある

### `todo -> delete`

- 完了した
- 不要になった
- 実行しない判断が確定した

#### `todo` のアーカイブを原則作らない理由

- 実行キューを履歴化すると探索コストが増えやすい
- 完了済み todo の参照価値は、知識層より低いことが多い
- 頻出パターンだけを `tips/` へ蒸留する方が軽くて自然である

### `playbook -> context`

- 家庭、特定相手、特定仕事など、個別条件への依存が強い
- 他者へ配る運用ルールというより、運用主体の固有制約メモとして読む方が自然である
- 一般ルールに見えるが、実際にはプロフィール依存で移植しにくい

### `theory -> playbook`

- 抽象論より運用ルールとして持つ方が自然である
- trigger / indicators / default_response に落とせる
- 抽象概念より即応性が重要である

### `playbook -> experiments`

- 即応ルールとしてはまだ再現性が弱い
- 日常運用の規則より、試行記録として読むほうが自然である
- 手元試行・外部接続・公開前提のいずれかで再検証したい

#### 戻し先の目安

- 仮説の鮮度を保ちたいなら `fragments`
- 手順として試行し直したいなら `experiments`
- 個別条件のメモとして保持したいなら `context`

### `theory -> experiments`

- 抽象構造としては見えているが、観測や応用先の検証が足りない
- 理論として固定するには早く、試行層へ戻したほうが更新しやすい
- まず internal / external / public のいずれかで挙動を確認したい

### `interfaces -> experiments`

- 他者向け形式としてはまだ安定していない
- 実際には公開物より試作品・接続試験の比重が高い
- 外部利用より先に、実験配置として摩擦を見たい

### `meta -> experiments`

- 継続運用の仕様というより、まだ試行的な管理方法である
- 正本・同期・判定基準として固定するには早い
- いったん実験層へ戻して検証したい
