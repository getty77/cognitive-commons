# Todo

このディレクトリには、
次に実行する項目を置く。

`todo/` は、
知識を保持する層ではなく、
実行キューを保持する層である。
ここでは、
残すことより回すことを優先する。
主語は、
`こうやる`
ではなく
`今回これをやる`
に置く。

## 役割

- `logs/`:
  状態差分、生観測、出来事記録
- `tips/`:
  すぐ効く軽量ハック、小さな運用知
- `todo/`:
  次にやる具体行動、実行キュー
- `playbook/`:
  概念層の運用ルール

## `todo/` に置くもの

- 今日やること
- 近いうちに試すこと
- `tips` を踏まえて実行したい行動
- `logs` の差分から起きた次アクション
- `tips` を見て起こした今回の実体

## 置かないもの

- 長期保存したい知識
- 背景説明や理論
- 完了後も参照価値が高いもの
- 単なる思いつきで、実行意図が薄いもの

## 運用原則

- `todo/` はガンガン消してよい
- 完了したら原則削除する
- 明示アーカイブは作らない
- 必要なら Git 履歴を参照する
- 頻出する todo や再利用価値が見えた todo だけを `tips/` へ蒸留する
- 優先順位は `差分が出たら logs`、`やり方として残すなら tips`、`どちらでもなければ delete` とする

## サブディレクトリ

- `private/`
  家庭、身体、感情、生活運用の todo
- `work/`
  顧客対応、会議、引き継ぎ、交渉など仕事運用の todo
- `dev/`
  開発環境、AI協働、実装ワークフローの todo

## 通常動線

通常利用では、

`logs diff -> tips参照 -> todo実行`

が主動線になる。

このとき、
`todo / tips / playbook` は
単純な昇格列ではなく、
`実践 / 手段 / 理論`
の近接三層として扱う。
通常は、
tip を参照して todo を起こし、
必要なら関連 playbook を見る、
というリンク運用を優先する。

また、
同じ束の中で処理した方が自然な todo 同士は、
同層リンクで結んでよい。
実行順、前提関係、同時処理のまとまりが見えるなら、
単独タスクとして孤立させない方が扱いやすい。

概念層である `fragments` や `playbook` は、
必要な時だけ見に行く。

また、
繰り返し出る todo は

`todo -> tips`

として蒸留してよい。
これにより、
todo 自体をアーカイブせずに、
再利用知だけを軽く残せる。
